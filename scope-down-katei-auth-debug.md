# Scope Down `katei-auth-debug`

This document lists the `katei-auth-debug` features that can be removed now that `test-runbook.md` assigns browser-side testing to Computer Use using the user's normal browser profile and saved credentials.

The goal is to stop maintaining a second browser automation stack inside the repo-local skill. After scope-down, the skill should either disappear entirely or shrink to a small non-browser helper for hidden API/state assertions that Computer Use cannot observe.

## New Ownership Boundary

Computer Use owns:

- Browser launch through the user's normal browser profile.
- Login using credentials already available in the browser.
- Navigation to `/boards`.
- Clicking visible UI controls.
- Typing into board/card forms.
- Selecting locales.
- Accepting confirmation dialogs.
- Screenshots and visible evidence.
- Cleanup of temporary boards/cards through visible UI.

`katei-auth-debug` should no longer own:

- Isolated Chrome profile management.
- Chrome DevTools Protocol connection management.
- Setting `HttpOnly` session cookies in Chrome.
- Browser navigation, clicking, typing, waiting, or screenshotting.
- Browser console, page error, failed-request, or selector snapshot collection.
- UI scenario automation for board lifecycle, localization, manual locale editing, workspace switching, or invite acceptance.

Potential remaining scope:

- Optional hidden API/state readback for assertions that are not visible in the UI.
- Optional local fixture data only if we still need deterministic non-production data.
- Optional app-level debug-auth smoke checks if kept outside the Computer Use browser flow.

## Remove Immediately

These features are fully superseded by `test-runbook.md` and can be deleted from the skill without losing a browser-side test capability.

### Isolated Chrome Launch

Remove:

- `scripts/run-chrome.sh`
- Config fields under `chrome`
- Constants in `scripts/lib/config.mjs`:
  - `DEFAULT_CHROME_BINARY_PATH`
  - `DEFAULT_REMOTE_DEBUGGING_PORT`
  - `DEFAULT_USER_DATA_DIR`
- Config normalization:
  - `normalizeChromeConfig`
  - `normalizePort`
- Artifact note about `/tmp/katei-auth-debug-profile`

Reason:

- Computer Use should use the normal browser profile so saved credentials are available.
- The old isolated profile works against the new assumption by requiring debug-cookie/session injection.

## Remove Or Rewrite With Readback Replacement

These features duplicate browser-side mutation flows, but they also currently provide hidden assertions. Remove the mutation parts and replace them with readback-only checks if those hidden assertions still matter.

### Direct API Localization Generation

Remove:

- `scripts/exercise-localization-flow.mjs`
- `latest-localization-flow.json`
- Direct `POST /api/workspace/localizations/generate` from the skill.
- Candidate mutation request construction for localization smoke runs.

Potential replacement:

- A read-only helper that, after Computer Use performs Scenario 4, fetches `/api/workspace` and verifies that a given `workspaceId`, `boardId`, `cardId`, and target locale now contain meaningful title/details.

Reason:

- Computer Use now triggers localization through visible UI.
- The old script mutates state through the API and bypasses the browser behavior the runbook is meant to validate.

Related code that can likely go with it:

- Most of `scripts/lib/localization_flow.mjs`
- Tests for `selectLocalizationCandidate` if no readback helper uses that selection logic
- Tests for `summarizeBoardLocalizationState` if no report uses it

### Direct API `review.origin` End-To-End Mutation

Do not keep the current shape long-term.

Current file:

- `scripts/verify-review-origin.mjs`

Remove from current behavior:

- API-side `card.create`
- API-side localization generation
- API-side `card.locale.upsert` after generation
- API-side `card.locale.discard`
- API-side manual locale recreation
- API-side card cleanup as the main scenario path

Potential replacement:

- A readback-only verifier that receives identifiers from the Computer Use run:
  - `workspaceId`
  - `boardId`
  - `cardId`
  - source locale
  - target locale
- It should fetch workspace state and assert:
  - Source variant `review.origin = human`
  - Generated target variant `review.origin = ai`
  - Generated provenance `includesHumanInput = false`
  - Manual edit after generation keeps `review.origin = ai`
  - Manual edit records human input
  - Discard removes target variant
  - Manual recreation sets `review.origin = human`

If readback after each browser step is required, create a smaller step-specific verifier instead of one script that performs the entire flow itself.

Reason:

- `test-runbook.md` Scenario 5 now owns the browser-visible review-origin flow.
- Hidden origin/provenance values are still not visible in normal UI, so only the assertion layer should remain.

Related code to reassess:

- `scripts/lib/review_origin_verification.mjs`
- `findReviewOriginVerificationTarget`
- Tests for target selection
- `smoke-review-origin.sh`

### Hosted Debug Auth For Browser Login

Remove from browser workflow:

- `debug-route` as the default way to obtain a browser session.
- `cookie` mode as a fallback way to inject `katei_session` into Chrome.
- `scripts/store-debug-secret-in-keychain.sh`
- Keychain fields in config:
  - `auth.secretKeychainService`
  - `auth.secretKeychainAccount`
- Cookie fields in config:
  - `auth.cookieName`
  - `auth.cookieEnvVar`
- Browser-session use of `obtainKateiSession`
- Browser-session use of `Network.setCookie`

Potential replacement:

- If any API readback helper remains, keep a small non-browser auth helper temporarily.
- Prefer an explicit API-readback auth story instead of making a browser skill manage debug secrets.

Reason:

- The new runbook assumes real browser login through saved credentials.
- Debug-auth session minting and cookie injection were only needed because the old skill controlled an isolated Chrome profile.

Important distinction:

- This does not mean removing `POST /__debug/login` from the Katei app. It only scopes it out of the browser-side skill. App-level debug auth may still be useful for server tests, fixture tests, or a separate API helper.

## Suggested Final Shape

Option A: remove the skill completely.

- Use `test-runbook.md` for browser testing.
- Add small standalone API verification scripts only when hidden assertions are needed.
- Remove `.agents/skills/katei-auth-debug` from repo-local skills.

Option B: shrink and rename the skill.

- New purpose: `katei-state-readback` or `katei-api-verify`.
- Keep no browser automation.
- Keep no CDP.
- Keep no Chrome config.
- Accept explicit identifiers from a Computer Use run.
- Produce compact hidden-state verification reports.

Recommended path:

1. First remove CDP/browser-driving scripts and config.
2. Keep `verify-review-origin.mjs` temporarily only as a safety net.
3. Replace `verify-review-origin.mjs` with readback-only checks keyed by Computer Use-created data.
4. Delete or rename the skill once hidden-state coverage has a new home.
